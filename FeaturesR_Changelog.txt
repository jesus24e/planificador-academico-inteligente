Cambios en la BD
---------------------------------------------------------------------------

[1] Archivo: database_helper.dart
    Ubicacion: lib/data/database/database_helper.dart
    Proposito:
      - Se anadio "await" a las sentencias execute de creacion de tablas
        (antes el "execute" se disparaba sin esperar, lo que podia provocar
        condiciones de carrera al inicializar la BD).
      - Se centralizaron constantes "dbName", "dbVersion" y "tableActividades"
        para evitar strings magicos repartidos en el codigo.
      - Se agrego "onConfigure" con "PRAGMA foreign_keys = ON" para dejar
        habilitadas las llaves foraneas.
      - Se agrego "onUpgrade" como gancho de migracion para evitar tener que
        desinstalar la app cuando se cambie el esquema en el futuro.
      - Se reforzo el esquema de la tabla "actividades":
          * "descripcion" pasa de "TEXT DEFAULT ''" a "TEXT NOT NULL DEFAULT ''".
          * "horas_dedicadas" pasa de "INTEGER DEFAULT 0" a
            "INTEGER NOT NULL DEFAULT 0 CHECK (horas_dedicadas >= 0)".
          * "tipo" recibe CHECK en ('tarea','examen','proyecto').
          * "prioridad" recibe CHECK en ('alta','media','baja').
      - Se crearon tres indices para acelerar consultas frecuentes:
          * idx_actividades_materia
          * idx_actividades_fecha_limite
          * idx_actividades_prioridad
      - Se agrego un metodo "close()" para poder cerrar la conexion de forma
        controlada cuando la app lo requiera.


[2] Archivo: activity_dao.dart
    Ubicacion: lib/data/database/dataAccessObject/activity_dao.dart
    Proposito:
      - Se corrigio un bug critico en "insert": antes se invocaba
        "db.database.insert(...)" (Database no expone ".database"); ahora se
        invoca correctamente "db.insert(...)". Esta era la causa por la que
        las inserciones individuales fallaban en runtime.
      - "getById" ahora retorna "Activity?" (nullable). Antes hacia
        "response.first" sin verificar y crasheaba si el id no existia.
      - "getAll" y "getBySubject" ahora ordenan por "fecha_limite ASC" para
        un orden estable y predecible en la UI.
      - Se agrego "getByDateRange" para consultar actividades por rango de
        fechas, util para el calendario y futuras vistas semanales/diarias.
      - Se agrego "insertMany" que inserta todo dentro de una transaccion
        usando "batch", lo cual es mucho mas rapido que N inserts sueltos.
      - Se agrego "count" para saber cuantos registros hay sin traerlos todos
        a memoria.
      - "deleteAll" ahora borra dentro de una transaccion y resetea el
        contador de AUTOINCREMENT de "sqlite_sequence", asi los ids vuelven
        a empezar desde 1.
      - "_toMap" ahora incluye el id solo si no es null.
      - "_fromMap" usa casts explicitos para mejor seguridad de tipos.
      - Se centralizo el nombre de la tabla en una constante "_table".

Creación de tabla "Materias" y uso en la aplicación
---------------------------------------------------------------------------

Se modelo la entidad faltante "Materia" (Subject), que la UI ya consumia
de forma hardcoded (MateriaCard con nombre/profesor/horario) pero que no
tenia respaldo en la base de datos. Las actividades quedan vinculadas a
las materias mediante una llave foranea, garantizando que no puedan
existir actividades referenciando una materia inexistente.

[11] Archivo: database_helper.dart  (MODIFICADO)
     Ubicacion: lib/data/database/database_helper.dart
     Proposito:
      - Se incremento "dbVersion" de 1 a 2.
      - Se anadio constante "tableMaterias".
      - "_createTables" ahora crea PRIMERO la tabla "materias" y DESPUES
        la tabla "actividades" (orden requerido por la FK):
          * materias.nombre es UNIQUE COLLATE NOCASE.
          * actividades.materia es FOREIGN KEY a materias(nombre) con
            ON UPDATE CASCADE (renombrar una materia se propaga a sus
            actividades) y ON DELETE RESTRICT (no se puede borrar una
            materia que tenga actividades, evita huerfanos).
      - Se implemento "_migrateToV2" para usuarios que ya tenian la BD
        en version 1: crea la tabla materias, la puebla con las materias
        DISTINCT que existian en actividades, recrea actividades con la
        FK siguiendo el patron estandar de SQLite (crear nueva, copiar,
        drop, rename) y reconstruye los indices. Todo dentro de una
        transaccion para garantizar atomicidad.
      - "actividades.materia" usa COLLATE NOCASE para que las
        comparaciones sean insensibles a mayusculas/minusculas.

Notas adicionales
-----------------
- La UI de la pestaña "Mis materias" (materias_tab.dart) sigue mostrando
  datos hardcoded como antes. La capa de datos para materias quedo lista
  para ser conectada cuando se decida tocar la UI; esta iteracion no
  modifico ningun archivo de UI.
- La migracion a v2 es no destructiva: usuarios con la app instalada
  conservan sus actividades y obtienen automaticamente una tabla materias
  con sus materias existentes.
- La FK con ON UPDATE CASCADE permite que en el futuro, si la UI deja
  renombrar una materia, todas las actividades asociadas se actualicen
  automaticamente sin codigo adicional.

Uso de filtros dinámicos.
---------------------------------------------------------------------------

Se rehizo la pestana "Mis tareas" para que lea de la base de datos en
lugar de la lista hardcoded, los filtros se calculen dinamicamente a
partir de los datos reales, las tareas se puedan abrir para ver detalle
y editar (incluyendo marcarlas como completadas/pendientes) y se valide
que las nuevas tareas referencien una materia existente.

Notas adicionales
-----------------
- Se omitio el parametro "locale" en showDatePicker porque MaterialApp
  no tiene flutter_localizations configurado; pasarle un locale sin esos
  delegates tiraria una excepcion en runtime. El picker funciona en
  ingles. Si se quiere localizar, basta agregar flutter_localizations al
  pubspec y los delegates al MaterialApp en una iteracion futura.
- No se modificaron home_screen, calendar_screen, priorityList ni
  activityCard. Estos siguen comportandose como antes; los nuevos
  campos de la entidad Activity son retro-compatibles porque
  "completada" tiene default false.
- "activityList" y "mapDateActivity" siguen existiendo en
  actividades_sim.dart porque otras pantallas (priorityList,
  calendar_screen) las consumen aun. Refactorizar esas pantallas no
  formaba parte de esta solicitud.
